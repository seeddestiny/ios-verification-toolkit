# 验证与探查报告

## 输出授权

- 默认在当前对话中输出 Markdown 报告，不自动写入外部系统。
- 只有用户明确要求外部文档时，才使用当时可用且已授权的文档能力创建。
- 不通过关闭 sandbox、扩大权限或绕过安全控制来生成报告；外部能力不可用时回退到当前对话。
- 创建外部文档前必须脱敏。完整 syslog、用户内容、token、账号、设备标识和本地绝对路径不得直接上传。

## 报告结构

```markdown
# <需求/改动名> 渐进式验证报告

## 概览
- 目标 diff/commit/工作树
- 依据的技术方案或现有业务语义
- 设备、系统、包类型
- 场景数：通过 / 修复后通过 / 阻塞 / 未覆盖

## 全局静态检查
- 检查范围与使用的仓库现有静态工具
- 发现的问题、真实业务修复和复查结果
- 门禁结论：通过 / 阻塞

## 全局本地基线编译
- 实际命令、scheme/target 和配置
- 基线编译结果；若有业务修复，记录修复后的重新编译结果
- 确认未执行依赖安装/更新、工程生成、bootstrap、签名更新或远程编译
- 门禁结论：通过 / 阻塞

## 运行时场景结果
### <场景名>
- 输入与前置状态
- 预期
- 操作路径
- 脱敏日志证据
- 结论
- 业务修复：根因、修改点、重验结果

## 未覆盖与限制
- 不可达分支及原因
- 环境失败或证据不足项
- 若有性能数据，说明样本数与口径

## 清理确认
- 本地持续日志采集已停止，临时全量日志文件已删除
- 每个场景的 VERIFY_/PROBE_、hardcode、mock 和环境变量门控均已清除
- 临时验证代码未 git add、未 commit
- 工作树只保留用户原有改动和真实业务修复
- 清理后静态残留扫描与最终本地编译已通过
```

## 证据规则

- 每条证据包含唯一 runId/scenario marker，避免混入旧日志。
- 只引用能证明判断的少量日志行，不复制整段 syslog。
- “未观察到”与“证明不存在”要区分；日志为空时不得直接宣称分支未执行。
- 不把场景日志命中描述成形式化代码覆盖率。
- 静态检查和本地基线编译各给出一个覆盖整个目标改动的全局结论，不按运行时场景重复；只有运行时结果按场景展开。后一个阶段不得掩盖前一个阶段的失败或未执行。
- 业务修复必须写明它仍处于原需求和原业务语义内；若改变语义，报告中标记为待用户确认而不是已修复。
