# iOS 渐进式代码验证与逻辑探查

## 模式与修改边界

- **验证模式**：对照技术方案、目标 diff/commit 和现有业务语义，验证分支、数据优先级、异常与兜底行为。若日志证明原业务逻辑存在缺陷，直接在原需求范围内修复并重验。
- **探查模式**：用临时埋点观察调用顺序、数据取值、缓存/分支命中和诊断耗时，不预设结论。
- **允许直接修复**：条件判断、数据处理、状态流转、异常处理等明显偏离既有设计的实现。
- **必须先确认**：改变产品语义、技术方案、公共接口、数据协议，扩大需求范围，或重构无关模块。

临时日志和 hardcode 只是验证设施，不是业务修复：**不得 `git add`、不得 commit、不得保留到下一个场景。每个场景结束、失败、阻塞或中断时都立即清理。**

## 步骤 0：确定范围并制定场景计划

1. 从当前工作目录和用户指定对象确定 Git 根目录、目标 diff/commit、技术方案或探查问题；不要绑定某个固定 checkout。
2. 执行 `git status --short`，记录用户已有改动；后续不得用 `git reset --hard`、`git checkout --` 等方式清理。
3. 验证模式要求目标代码确实包含需运行时验证的复杂逻辑。目标改动可以来自当前会话、已有工作树、分支或 commit，不要求由本 Agent 编写。
4. 仅为运行时阶段把可达且与改动相关的分支拆成场景矩阵。记录场景名、输入/AB/数据状态、预期、操作路径、证据字段和未覆盖原因。静态检查与基线编译不得按此矩阵重复拆分；不要把日志命中宣称为形式化代码覆盖率。
5. 性能需求先区分“诊断计时”和“正式基准”。单次日志只用于定位；正式结论需预热、多次采样和 Instruments/os_signpost 等工具。
6. 为本轮生成唯一 ASCII `runId`（只含字母、数字、点、下划线或连字符），为每个场景生成 marker：验证用 `VERIFY_<runId>_<场景>`，探查用 `PROBE_<runId>_<主题>`。

## 渐进式门禁（硬性顺序）

始终按以下顺序推进：

```text
全局静态检查（整个目标 diff/commit）
→ 全局本地基线编译（整个当前业务基线）
→ 运行时场景 1 → 清理场景 1 临时设施
→ 运行时场景 2 → 清理场景 2 临时设施
→ ...
```

- 全局静态检查和全局基线编译以整个目标改动为单位，不按运行时场景重复执行；只有运行时检查按场景拆分。
- 前一阶段未通过时停止后续阶段，不得用运行时结果替代静态检查或编译结果。
- **静态检查和基线编译完成前，不得发现/选择设备，不得启动日志、App 运行或 UI 自动化工具，也不得启动 Appium/WDA/隧道或注入运行时 hardcode/日志。**
- 环境阻塞与代码失败分开报告；不得为了进入运行时阶段而跳过失败门禁。
- 运行时阶段产生真实业务修复时，当前全局门禁结果立即失效。先清理当前场景全部临时设施，只保留真实业务修复；再对新的完整业务基线重新执行全局静态检查和全局本地基线编译。通过后更新运行时场景矩阵，重新执行当前场景、受该修复影响的已通过场景以及剩余场景。

## 阶段 1：全局静态检查

1. 一次性覆盖整个目标 diff/commit、用户指定范围及其必要上下游；不得按运行时场景只检查局部文件。对照技术方案、现有接口契约和业务语义检查分支完整性、数据优先级、状态流转、空值/边界、线程与生命周期、异常/兜底、资源清理和兼容性。
2. 执行与范围匹配的只读或静态工具检查：至少对目标工作树或 commit range 执行对应的 `git diff --check`；仓库已有 lint、格式或静态分析入口时使用现有入口，不临时引入新工具或全局依赖。
3. 搜索目标范围内遗留的验证 marker、临时 hardcode、调试日志、敏感信息和明显不可达代码。初始静态检查不得把本轮尚未注入的运行时设施计入结果。
4. 发现原业务逻辑范围内的明确问题时直接修复并从本阶段重新检查；涉及产品语义、公共接口、协议、范围扩大或无关重构时先请求确认。
5. 输出一个全局静态检查结论和未决项，不为每个运行时场景重复生成静态结论。只有整个目标范围无阻断问题时，标记全局静态门禁通过并进入阶段 2。

## 阶段 2：全局本地基线编译

1. 只从当前仓库已存在的 `.xcworkspace` / `.xcodeproj`、scheme、configuration 和构建说明中确定入口。优先直接使用本地 `xcrun xcodebuild` / `xcodebuild`；仅当仓库已有命令经检查确认只执行本地编译时才可使用。不得引用其他项目的工具、目录或命令。
2. 在工作树不含本轮任何运行时日志/hardcode 的前提下，按 `references/build-and-logs.md` 执行能覆盖整个目标改动的本地 Debug 构建。全局门禁使用 `generic/platform=iOS`，不要求提前选择真机；只形成一个全局基线编译结论，不按场景分别编译基线。
3. **禁止准备或修复依赖环境**：不得执行 `pod install/update`、`bundle install`、Swift Package resolve/update、npm/yarn/pnpm install、brew/gem 安装、工程生成、bootstrap、`xcodebuild -resolvePackageDependencies` 或任何等价步骤；不得修改 `Podfile.lock`、`Package.resolved`、Pods 或工程生成产物。现有工程/依赖/签名不完整时按环境阻塞报告。
4. 源码编译错误在原范围内修复后，全局静态门禁失效；回到阶段 1 覆盖完整目标范围重新静态检查，再重新执行全局基线编译。依赖、签名、生成步骤或参数缺失按环境阻塞报告，不得当成业务失败，也不得切换远程/云端编译。
5. 记录一组覆盖整个目标改动的实际命令、scheme/target、配置和结果。只有全局基线编译成功，才能进入阶段 3。

## 阶段 3：运行时前置检查

本阶段在两项全局门禁通过后执行一次设备选择。日志采集、App 构建运行与 WDA UI 自动化是三个独立生命周期；不要因为只需日志或启动 App 而启动 MCP/Appium/WDA。

1. **默认三个工具已安装；跳过独立可用性检查。**不得预先执行 import/status/doctor、运行安装器、遍历 MCP tools，或创建全局 `ios_ui_automation` 注册。直接执行场景所需的首次真实命令，失败时再按具体层级报告。
2. 先用系统 Python 执行共享的 `mcp_server/device_discovery.py list`，读取**当时实际连接**的物理 iPhone/iPad。它不加载 MCP/Appium/WDA，也不属于日志或 App 工具。零台停止；单台自动选择；多台且用户未指定时，停止后续操作并按以下结构询问：
   ```text
   检测到多台当前可用的 iPhone/iPad，需要你选择本轮验证设备：
   1. <设备名称>（<型号>，<系统版本>，<连接方式>）
      硬件 UDID：<udid>
   2. ...
   请回复序号、设备名称或硬件 UDID；收到你的选择前，我不会启动真机验证。
   ```
   禁止写死默认 UDID、静默选择第一台，或在未收到选择时继续启动任何运行时工具。
3. 记录同一设备返回的硬件 UDID 与 CoreDevice identifier：`xcodebuild` 真机 destination、`idevicesyslog` 和 WDA 使用硬件 UDID；`devicectl` 优先使用 CoreDevice identifier，也可使用其明确支持的当前设备选择器。
4. 根据场景选择最小工具集合：仅日志判定时只使用日志工具；只需构建/安装/启动时只使用 App 工具；确需截图、UI 树或点击时才启动 `ios_ui_session.py` 和隧道。UI 工具启动后复用已有 Appium/WDA；WDA 仅在真实 session 明确缺失时首次安装，检测失败不得当作缺失。
5. 任一真实调用失败时，只停止依赖它的后续动作，清理已启动的工具和临时设施，并报告失败属于日志、App 本地工具或 WDA UI 工具中的哪一层。不得自动运行安装器、依赖更新、doctor、升级或全局注册。
6. 为即将执行的场景建立“临时注入清单”，包含文件、marker、hardcode 条件和插入位置。三个工具的命令、设备参数和退出清理见 `references/runtime-tools.md`；UI MCP 细节见 `references/mcp-automation.md`。

## 单场景闭环

全局阶段 1、2 均通过后，才把运行时验证按场景独立执行以下循环；本场景通过或无法继续后，先完成清理，再进入下一个场景。场景内为临时验证代码生成装机包的编译属于运行时执行步骤，不是阶段 2 的全局基线编译，也不产生按场景拆分的“基线编译结论”。

### 1. 注入可撤销的临时验证代码

- 埋点前必读 `references/temporary-validation-code.md`。
- 所有临时代码放在当前仓库已有的本地调试编译门控中；没有项目自定义门控时使用 `#if DEBUG`。不得发明或依赖产品私有宏；**编译门控不能单独作为 hardcode 开关**。
- hardcode 和场景日志必须再受运行时环境变量 `IOS_CODE_VERIFY_SCENARIO=<场景>` 控制，避免代码存在期间污染其他调试启动。
- 复用当前模块已知可见/已在白名单中的日志 tag，把 marker 写入 `items` 正文。`logError` 同样受 tag 白名单过滤，禁止用新 `VERIFY_` tag 直接赌白名单。
- 日志携带能判定行为的关键字段和 runId，不记录 token、用户内容、账号信息或不必要的完整业务数据。

### 2. 构建与装机

- **只允许本地编译（硬性）**：使用独立 App 工具包装的本机 `xcodebuild`，或使用经检查确认只做本地编译的仓库现有命令；禁止执行任何依赖安装/更新、工程生成、bootstrap、`-allowProvisioningUpdates` 或远程/云端编译。详见 `references/runtime-tools.md` 与 `references/build-and-logs.md` 第 1 节。
- 这里编译的是已通过基线编译、但包含本场景临时验证设施的本地调试包；仍要先做语法/最小编译检查，再构建装机包。
- **编译失败**：若错误来自临时埋点/hardcode，修复临时设施后重试本场景装机包，不据此修改业务逻辑；若暴露真实业务源码错误，修复后先执行本场景强制清理，只保留真实业务修复，再让阶段 1、2 的全局门禁覆盖新的完整业务基线，通过后重新注入并构建本场景。环境问题按环境错误处理，不当作业务失败，也不因此切远程。
- 真机目标使用本轮动态选择的硬件 UDID 执行本地编译；使用 App 工具的 `install` 安装产物。此时默认尚未启动 WDA，避免安装被 UI session 干扰；纯编译不启动或停止隧道。
- 构建失败时先区分源码错误、签名/安装错误和环境问题，不把环境失败判成业务逻辑失败。

### 3. 在操作前启动日志采集并启动 App

1. 用轻量日志工具 `start --session <runId>-<scene>` 启动 `idevicesyslog`；这一步不得启动 MCP、Appium、WDA 或隧道。
2. 若场景需要 UI 操作，此时才启动按需 UI MCP 并确保隧道就绪；纯日志场景跳过。
3. 用独立 App 工具的 `launch --terminate-existing` 启动目标 App，注入 `IOS_CODE_VERIFY_RUN_ID` 与 `IOS_CODE_VERIFY_SCENARIO`。不得把 token、账号或其他凭据传给启动工具。
4. 三个工具分别判断成功；日志或 App 工具失败时不得归因于 WDA，UI MCP 失败也不得推翻已经完成的本地编译或日志工具结论。

### 4. 逐步操作与收集证据

- 需要 UI 时，每次只通过 UI MCP 执行一步：`screenshot` → 理解界面 → `tap` → 再次 `screenshot`；找不到元素时用 `get_ui_hierarchy`。不需要 UI 时不启动 WDA。
- 走完场景后用日志工具 `stop --contains <marker> --delete-file` 返回最小证据并删除全量文件。
- 若流程中断或任一工具报错，也在退出路径单独停止日志 session；UI MCP 是否存在不影响日志清理。
- 日志为空时依次检查：采集是否在操作前启动、marker 是否一致、tag 是否在白名单、目标代码是否进入、构建产物是否为本轮版本。不得直接判定“分支未执行”。

### 5. 判断、修复并重验

- 日志符合预期：标记场景通过。
- 日志证明业务实现偏离既有设计：定位根因，在原业务逻辑范围内直接修复；立即停止采集并清理本场景全部临时验证设施，只保留真实业务修复。随后让新的完整业务基线重新通过阶段 1 全局静态检查和阶段 2 全局本地基线编译，再重新注入本场景设施、构建装机包并重验。循环直到全局静态、全局编译和运行时场景均通过。
- 证据不足：补充最小必要日志后重验，不凭猜测修改业务代码。
- 修复需要改变产品语义、公共接口、协议或扩大范围：停止并请求确认。

### 6. 场景级强制清理

无论场景通过、失败、阻塞还是中断，都执行：

1. 用独立日志工具停止并删除本场景采集文件；若启动过 UI MCP，再单独停止该会话并恢复隧道状态。
2. 根据“临时注入清单”精确删除本场景全部 `VERIFY_`/`PROBE_` 日志、环境变量判断、hardcode、mock 返回值及辅助类型；保留真实业务修复。
3. 在本场景涉及的文件中搜索 marker 和 `IOS_CODE_VERIFY_RUN_ID` / `IOS_CODE_VERIFY_SCENARIO`，确认零残留。
4. 对照场景开始前的用户改动，确认没有误删或还原用户代码。
5. 对撤销临时设施后的源码做语法/最小编译检查，确认清理没有破坏构建；它属于运行时场景清理，不生成或替代阶段 2 的全局基线编译结论。清理未完成时不得进入下一场景。
6. **任何 commit 前再次扫描所有待提交文件；发现验证 marker、临时 hardcode 或探查日志时禁止 commit。**

## 整轮结束

- 按“一个全局静态检查结论 → 一个全局基线编译结论 → 多个运行时场景结果”汇总真实业务修复和未覆盖项，不得为每个场景重复生成静态/基线编译结论。
- 无论成功、失败、阻塞或中断，都分别停止所有日志 session；仅在本轮启动过 UI MCP 时执行 `ios_ui_session.py stop --session <runId>`，恢复隧道初始状态并关闭 MCP。日志工具自身有最长 30 分钟的采集兜底，不依赖 MCP 退出清理。
- 默认在当前对话输出报告；只有用户明确要求时才创建外部文档。详见 `references/report-output.md`。
- 最终再次执行静态残留扫描和覆盖真实业务修复的本地编译，再检查工作树：只允许保留用户原有改动和经授权的真实业务修复，不得残留任何验证/探查设施。
